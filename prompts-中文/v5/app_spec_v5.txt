  <project_name>PodcastBrain — v5：完整功能集（準最終版）</project_name>

  <overview>
    PodcastBrain v5 是完整的準最終版規格，在 v4 的基礎上新增最後一項主要
    功能——批次處理佇列（Batch Processing Queue）。使用者現在可以一次貼入
    多個 URL（每行一個）進入批次佇列，應用程式會依序處理這些 URL，並以
    即時狀態表格顯示每個節目在「下載 → 轉錄 → 分析」流程中的進度。
    佇列狀態會持久化到 SQLite，因此重新載入頁面不會遺失排隊中的任務。
    當批次處理完成時，可透過 smtplib 發送可選的電子郵件通知。
    Plotly 渲染的章節時間軸視覺化圖表已新增至「章節」分頁。
    本版本功能完整，與最終版 app_spec.txt 的功能覆蓋範圍相同，
    差異僅在於提供給自主程式碼代理的實作細節層級與複雜度指引。
  </overview>

  <core_features>
    1. 音訊輸入 — URL 文字輸入（YouTube 觀看 URL、直連 MP3/M4A/WAV 連結）與檔案上傳
       小工具（.mp3、.m4a、.wav、.ogg）。yt-dlp 將音訊下載至暫存目錄。進度條
       透過解析 yt-dlp stdout 更新。取消按鈕可終止子程序。

    2. 本地 Whisper 轉錄 — 模型選擇下拉選單（tiny/base/small/medium/large），附每個
       模型的預估時間與準確度說明。Whisper 透過 Python API 呼叫：
       import whisper; model = whisper.load_model(model_name);
       result = model.transcribe(audio_path, language=language)。
       提取：result["text"]（完整逐字稿）、result["segments"]（含 start/end/text 的清單）。
       根據音訊時長與已用時間估算進度。
       處理：找不到檔案、不支援的格式、模型下載失敗、CUDA/CPU 選擇。

    3. Claude 章節偵測 — Claude 接收完整逐字稿文字並回傳 JSON 陣列：
       [{title, start_seconds, end_seconds, summary}, ...]。驗證 JSON 結構。
       短集數（不足 5 分鐘）的退回方案：涵蓋完整時長的單一章節。指數退避重試 3 次。
       章節儲存於資料庫並以可點擊的時間軸顯示。

    4. Claude 完整分析 — 三次獨立的 Claude API 呼叫：
       (a) generate_summary_quotes_actions(transcript_text, api_key) — 回傳 JSON，包含：
           summary（3-5 段含關鍵主題的摘要）、quotes（{text, speaker, timestamp_seconds} 陣列）、
           action_items（具體後續步驟的編號清單）。
       (b) identify_speakers(transcript_text, api_key) — 回傳含說話者標籤的 JSON。
       所有呼叫使用重試邏輯（3 次，指數退避）。將合併結果以 JSON 格式儲存
       於 episodes.claude_analysis 欄位中。

    5. 向節目提問 Q&amp;A — 基於關鍵字的片段檢索找到相關的逐字稿部分。
       Claude 嚴格依據檢索到的片段回答——「僅引用逐字稿」系統提示。
       引用模式顯示使用的確切逐字稿段落。Q&amp;A 記錄依節目儲存於資料庫。
       使用 st.chat_input 與可捲動的 st.chat_message 歷史記錄呈現聊天介面。

    6. 匯出選項 — Markdown（.md）含完整結構（摘要、章節、引言、逐字稿），
       PDF（reportlab，含章節標頭格式）、SRT 字幕檔（來自 Whisper 片段）、
       純文字 TXT（僅逐字稿）、包含所有格式的 ZIP 壓縮檔。全部透過 st.download_button 提供。

    7. 附搜尋功能的節目資料庫 — 透過 st.text_input 對所有節目逐字稿進行 FTS5 全文搜尋。
       以日期範圍（st.date_input）與時長範圍（st.slider，0-180 分鐘）篩選。
       節目卡片網格（3 欄），含縮圖、標題、時長、處理日期。點擊卡片
       開啟節目檢視器。snippet() 函式在搜尋結果中提供高亮摘錄。

    8. 批次處理佇列 — 透過 st.text_area（每行一個 URL）將多個 URL 加入佇列。
       「Add to Queue」按鈕插入 status='queued' 的 Episode 記錄。依序處理
       （一次一個，避免資源競爭）。狀態表格顯示：url、title、status、
       progress、eta。狀態值：queued/downloading/transcribing/analyzing/complete/failed。
       使用 st.empty() + st.rerun() 每 5 秒自動重新整理表格。批次完成時
       可透過 smtplib 選擇性發送電子郵件通知。佇列透過 SQLite 跨頁面重新載入持久化
       （episodes 資料表的 status 欄位）。
  </core_features>

  <database_schema>
    -- 主要節目記錄
    CREATE TABLE episodes (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        title TEXT,
        source_url TEXT NOT NULL,
        source_type TEXT CHECK(source_type IN ('youtube','direct_url','file_upload')),
        audio_path TEXT,                  -- 已下載音訊檔案的暫存路徑
        duration_secs INTEGER,
        whisper_model TEXT,               -- 例如 'base', 'small'
        language TEXT DEFAULT 'en',
        status TEXT CHECK(status IN ('queued','downloading','transcribing','analyzing','complete','failed')),
        error_message TEXT,
        claude_analysis TEXT,             -- JSON: {summary, quotes, action_items, speakers}
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
        processed_at TIMESTAMP
    );

    -- 含片段級時間戳的完整逐字稿
    CREATE TABLE transcripts (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        episode_id INTEGER NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
        full_text TEXT NOT NULL,
        segments TEXT,                    -- JSON 陣列：[{start, end, text}, ...]
        word_count INTEGER,
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );

    -- 來自 Claude 分析的章節標記
    CREATE TABLE chapters (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        episode_id INTEGER NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
        title TEXT NOT NULL,
        start_seconds INTEGER NOT NULL,
        end_seconds INTEGER,
        summary TEXT,
        sort_order INTEGER NOT NULL
    );

    -- 每個節目的 Q&A 互動記錄
    CREATE TABLE qa_history (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        episode_id INTEGER NOT NULL REFERENCES episodes(id) ON DELETE CASCADE,
        question TEXT NOT NULL,
        answer TEXT NOT NULL,
        source_ts TEXT,                   -- 使用的 [start, end] 時間戳對的 JSON 陣列
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    );

    -- FTS5 虛擬資料表，用於逐字稿全文搜尋
    -- 在儲存逐字稿後透過觸發器或手動插入來填充
    CREATE VIRTUAL TABLE transcripts_fts USING fts5(
        episode_id UNINDEXED,
        full_text,
        content='transcripts',
        content_rowid='id'
    );

    -- 保持 FTS5 同步的觸發器
    CREATE TRIGGER transcripts_ai AFTER INSERT ON transcripts BEGIN
        INSERT INTO transcripts_fts(rowid, episode_id, full_text)
        VALUES (new.id, new.episode_id, new.full_text);
    END;
  </database_schema>

  <key_interactions>
    流程 1 — 處理新節目：
      使用者輸入 YouTube URL → 點擊「Process Episode」→ st.status() 顯示即時更新：
      「下載音訊中...」、「轉錄中（可能需要幾分鐘）...」、「使用 Claude 分析中...」
      → 節目檢視器出現，所有 4 個分頁均已填入資料 → 匯出按鈕可見。

    流程 2 — 向節目提問：
      使用者點擊 Q&amp;A 分頁 → 輸入問題 → 點擊「Ask」→ qa_engine.py 依關鍵字搜尋
      逐字稿片段 → Claude 僅根據檢索到的段落回答 → 回應出現，
      附有「來源：」引用於 st.expander → Q&amp;A 對加入可捲動的聊天記錄。

    流程 3 — 搜尋節目資料庫：
      使用者點擊「Episode Library」→ 輸入搜尋查詢 → FTS5 執行 → 帶高亮片段的
      匹配節目卡片 → 點擊卡片從資料庫開啟完整節目檢視器。

    流程 4 — 批次處理多個 URL：
      使用者點擊「Batch Queue」→ 貼上多個 URL（每行一個）→ 點擊「Add to Queue」→
      點擊「Start Processing」→ 狀態表格每 5 秒更新一次，顯示每個節目的
      進度 → 完成後寄送電子郵件通知（若已設定）。
  </key_interactions>
